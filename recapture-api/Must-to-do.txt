AWS S3 Setup Checklist (manual — 10 minutes)
1. Create the two buckets
Go to S3 Console → Create bucket
Bucket 1 — msxr-raw-captures

Region: ap-south-1 (Mumbai) — matches Render region: singapore reasonably well; pick whichever is closer to your primary user base and keep both buckets + Render in the same region to avoid cross-region transfer costs
Block all public access: ON (keep default — never make this public)
Bucket Versioning: Off (not needed for raw captures)
Default encryption: SSE-S3 (default, fine)

Bucket 2 — msxr-model-artifacts

Same region as above
Block all public access: ON (CloudFront will access via Origin Access Control, not public ACLs)
Versioning: Off
Default encryption: SSE-S3 (default)

2. Configure CORS on msxr-raw-captures
This bucket needs CORS so the mobile app can upload directly via presigned URLs.

Go to bucket → Permissions → Cross-origin resource sharing (CORS) → paste:
json[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["PUT", "GET", "HEAD"],
    "AllowedOrigins": ["*"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
}
]

Tighten AllowedOrigins later if the mobile app ever runs uploads from a known web domain — for native mobile apps "*" is standard since there's no browser origin to restrict.

3. Configure lifecycle rule on msxr-raw-captures
Go to bucket → Management → Create lifecycle rule:

Rule name: expire-raw-captures
Scope: entire bucket
Action: Expire current versions of objects
Days after object creation: 90 (matches PRD: "retain 30–90 days")

Go to IAM Console → Users → Create user

Name: recapture-api-service
No console access — programmatic access only

2. JWT Middleware + Token Refresh Logic
The requireAuth middleware (verifies access tokens) already exists from the scaffold. What's missing is the refresh flow — short-lived access tokens + long-lived refresh tokens, with rotation.

Strategy
Two-token model:

Access token — short-lived (15 min), sent in Authorization: Bearer header, used for all API calls. Stateless — verified via JWT_SECRET signature only, no DB lookup.
Refresh token — long-lived (30 days), stored in MongoDB (hashed), used only to mint new access tokens via POST /auth/refresh. Stateful — DB lookup required, can be revoked.
This matches the PRD's acceptance criteria: "Token expiry & refresh supported."

New MongoDB collection: refreshTokens
typescript
interface IRefreshToken {
  _id: ObjectId;
  userId: ObjectId;
  tokenHash: string;        // SHA-256 hash of the raw refresh token — never store raw
  expiresAt: Date;           // 30 days from issue
  revokedAt?: Date;          // set on logout or rotation
  deviceInfo?: string;       // optional — "iPhone15,2 / iOS 17.2" for session management
  createdAt: Date;
}
// Indexes:
//   { tokenHash: 1 } unique
//   { userId: 1, revokedAt: 1 }
//   TTL index on expiresAt (auto-delete expired tokens)
Token issuance flow (on OTP verify, POST /auth/verify-otp)
Verify OTP
Generate access token: jwt.sign({ userId, authUid }, JWT_SECRET, { expiresIn: '15m' })
Generate refresh token: random 64-byte hex string (crypto.randomBytes(64).toString('hex'))
Hash the refresh token (crypto.createHash('sha256').update(token).digest('hex')) and store the hash in refreshTokens with 30-day expiresAt
Return both tokens to the app: { accessToken, refreshToken, expiresIn: 900 }
Refresh flow (POST /auth/refresh)
App sends { refreshToken } in request body (not as a header — refresh tokens are opaque, not JWTs)
Hash the incoming token, look up refreshTokens by tokenHash
Checks: token exists, not revoked, not expired
Rotate: revoke the old refresh token (revokedAt: now), issue a brand new refresh token + new access token
Return new pair: { accessToken, refreshToken, expiresIn: 900 }
Why rotation matters: if a refresh token is ever stolen, it can only be used once before the legitimate client's next refresh attempt fails (because the token was already rotated) — alerting you to compromise. This is the OWASP-recommended pattern for mobile refresh tokens.

Logout flow (POST /auth/logout)
App sends { refreshToken }
Hash it, find the record, set revokedAt: now
Access token remains valid until its 15-min expiry (stateless — can't be revoked early without a blocklist, which is usually not worth the complexity for a 15-min window)
Edge cases to design for
Expired refresh token → 401, app must route user back to AuthScreen (re-login via OTP)
Revoked/reused refresh token → 401 + optionally revoke all refresh tokens for that userId (defense against token theft — "revoke all sessions on suspicious reuse")
Clock skew → JWT exp claims have a small leeway (jwt.verify(token, secret, { clockTolerance: 10 })) to handle minor client/server time drift
Multiple devices → a user can have multiple active refresh tokens simultaneously (one per device) — don't revoke other devices' tokens on a normal refresh, only on explicit logout or theft detection
Mobile app side (Flutter)
AuthNotifier (from the P1 Flutter prompts, not yet generated) stores both tokens in Hive (boxSession). Dio interceptor: on 401 response, call /auth/refresh with the stored refresh token, update both tokens, retry the original request. If refresh also fails → clear session, route to AuthScreen.
